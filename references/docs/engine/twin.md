## Diagram Geometry Rules

The twin's master diagram is the product at the code level, drawn from the product's perspective, not a raw repo tree.

**Single products (product-local `osis/twin.md`):**
- Product name at top
- Owned systems as peers underneath
- Shared packages and infrastructure omitted unless they are load-bearing product surfaces
- Box-drawing characters (│ ├ └) to show hierarchy
- Readable in 2 to 3 minutes total

**Monorepos (root `osis/twin.md`):**
- Org name at top
- Products as peers
- Owned systems beneath each product
- Shared packages omitted
- Use `osis.json.products` map to identify product paths

**Altitude rules:**
- Keep it compressed. If a section reads like code documentation, cut it. If it reads like a product pitch, cut it. Describe reality.
- No implementation details (functions, variable names, library versions) unless they shape the product's surface.
- Skip systems that are vestigial or non-load-bearing.
